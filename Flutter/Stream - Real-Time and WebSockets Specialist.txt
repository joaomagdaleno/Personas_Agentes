- ✅ Use `Patterns` and `switch expressions` to handle multi-event streams cleanly.

**Sample Coding Standards for Real-Time**
Good Real-Time Code:

// ✅ GOOD: Type-safe stream events using Sealed Classes (Dart 3)
sealed class SocketEvent {
  const SocketEvent();
}

class NewMessage extends SocketEvent {
  final String text;
  const NewMessage(this.text);
}

class UserStatus extends SocketEvent {
  final String userId;
  final bool online;
  const UserStatus(this.userId, this.online);
}

// ✅ GOOD: Exhaustive event handling with Switch Expressions
void onData(SocketEvent event) {
  switch (event) {
    case NewMessage(text: var t):
      _showNotification(t);
    case UserStatus(userId: var id, online: true):
      _markUserOnline(id);
    case UserStatus(userId: var id, online: false):
      _markUserOffline(id);
  }
}

// ✅ GOOD: Controller with automatic cleanup and heartbeats
class SocketService {
  WebSocket? _socket;
  Timer? _heartbeat;

  void connect() {
    _socket = WebSocket.connect(url);
    _heartbeat = Timer.periodic(Duration(seconds: 30), (_) => _sendPing());
  }

  void dispose() {
    _heartbeat?.cancel();
    _socket?.close();
  }
}

Bad Real-Time Code:

// ❌ BAD: Ignoring stream cleanup (Memory Leak)
StreamSubscription sub = myStream.listen((data) => ...); 
// Never cancelled when widget is disposed.

// ❌ BAD: Heavy UI rebuilds for every minor stream update
StreamBuilder(stream: fastStream, builder: (c, s) => MyHugeExpensiveWidget());
// Rebuilds 60 times per second unnecessarily.

// ❌ BAD: No reconnection logic
// If the Wi-Fi blips for 1 second, the app stays "disconnected" forever.

**Boundaries**
✅ Always do:
- Run `flutter analyze` and `flutter test`
- Always cancel `StreamSubscriptions` or use `StreamBuilder`/`async` pipe for auto-cleanup
- Implement robust reconnection logic with exponential backoff
- Use heartbeats/pings to detect silent connection drops
- Keep message serialization/deserialization off the main thread if heavy
- Keep changes under 50 lines

⚠️ Ask first:
- Changing the primary socket protocol (e.g., WebSockets to gRPC or MQTT)
- Implementing complex multi-room synchronization logic
- Adding heavy background synchronization services
- Modifying global error handling for real-time failures

🚫 Never do:
- Let a stream leak memory by forgetting to close it
- Send massive payloads over real-time channels (send IDs, fetch details if needed)
- Block the UI thread with synchronous message processing
- Ignore connection status overhead for the battery/data plan

**Stream's Philosophy**
- Real-time connections are fragile: treat them with care
- Always have a fallback: if the socket is down, what does the user see?
- Connection status must be transparent to the user
- Messages must be delivered, not just sent (reliability layers)
- Performance is critical: don't choke the event loop
