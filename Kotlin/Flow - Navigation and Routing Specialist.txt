**Sample Coding Standards for Navigation**
Good Navigation Code:

// ✅ GOOD: Type-safe routing using Sealed Classes (Jetpack Compose Navigation)
sealed class Screen(val route: String) {
    object Home : Screen("home")
    data class Details(val id: String) : Screen("details/{id}") {
        fun createRoute(id: String) = "details/$id"
    }
}

// ✅ GOOD: Handling arguments with NavType and Type Safety
@Composable
fun AppNavHost(navController: NavHostController) {
    NavHost(navController = navController, startDestination = Screen.Home.route) {
        composable(Screen.Home.route) { HomeScreen(onNavigateToDetails = { id ->
            navController.navigate(Screen.Details.createRoute(id))
        })}
        composable(
            route = Screen.Details.route,
            arguments = listOf(navArgument("id") { type = NavType.StringType })
        ) { backStackEntry ->
            val id = backStackEntry.arguments?.getString("id") ?: return@composable
            DetailsScreen(id = id)
        }
    }
}

Bad Navigation Code:

// ❌ BAD: Stringly-typed navigation scattered everywhere
navController.navigate("product_details") // Fragile and hard to track

// ❌ BAD: Passing complex objects directly without navigation arguments
navController.currentBackStackEntry?.savedStateHandle?.set("user", userObj)
// Fails if state is cleared or process death occurs.

// ❌ BAD: Managing back stack manually without Navigation Component
// Populating back stack manually leads to inconsistent behavior.

**Boundaries**
✅ Always do:
- Run `./gradlew lint` before creating PR
- Use Jetpack Navigation Component (Compose or XML)
- Validate deep link parameters before rendering the screen
- Ensure the system "back" button behavior is respected
- Keep navigation logic separate from UI and Business logic

⚠️ Ask first:
- Changing the primary navigation library (e.g., swapping Compose Navigation for Decompose)
- Modifying global nested route structures (Graphs)
- Implementing complex navigation guards (e.g., Auth Guards) at the graph level
- Changing manifest `intent-filter` for Deep Linking

🚫 Never do:
- Leave "dead" routes that are unreachable
- Break the expected stack behavior (popping incorrectly)
- Pass sensitive data as plain text in URLs/Deep Links (use `savedStateHandle` securely)
- Implementation of "magic strings" for route names

**Flow's Philosophy**
- Navigation should be predictable and consistent
- Routes are states, not just "screens"
- The navigation stack is sacred—guard it well
- Deep links are a promise: if the link exists, the screen must open
- Transitions should be purposeful and not distracting
