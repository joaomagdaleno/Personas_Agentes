**Modern Coding Standards (Kotlin 2.0+)**
- ✅ Use `sealed classes` for defining exhaustive storage states (Loading, Success, Error).
- ✅ Use `Flow` and `StateFlow` for observing database changes asynchronously.
- ✅ Leverage `Data Classes` for structured data mapping to database entities.

**Sample Coding Standards for Storage**
Good Storage Code:

// ✅ GOOD: Repository pattern with intelligent caching (Kotlin Flow)
class ProductRepository(
    private val localDb: AppDatabase,
    private val api: ApiClient
) {
    fun getProducts(): Flow<List<Product>> {
        return localDb.productDao().getAll()
            .onEach { localProducts ->
                if (localProducts.isEmpty()) {
                    refreshFromApi() // Trigger fetch if empty
                }
            }
    }

    private suspend fun refreshFromApi() {
        try {
            val remote = api.fetchProducts()
            localDb.productDao().insertAll(remote)
        } catch (e: Exception) {
            // Handle error, maybe show cached data if available
        }
    }
}

// ✅ GOOD: Using Room with Coroutines and Flow
@Dao
interface UserDao {
    @Query("SELECT * FROM user WHERE id = :id")
    fun getUserById(id: String): Flow<User?>
}

// ✅ GOOD: Using Proto DataStore for type-safe key-value pairs
val Context.settingsDataStore: DataStore<Settings> by dataStore(
    fileName = "settings.pb",
    serializer = SettingsSerializer
)

Bad Storage Code:

// ❌ BAD: Blocking the Main Thread with heavy synchronous I/O
val data = File("data.json").readText() // Causes ANR

// ❌ BAD: Storing sensitive data in plain SharedPreferences
sharedPrefs.edit().putString("token", "secret").apply() // Vulnerable on rooted devices

// ❌ BAD: No migration strategy
// App crashes when you add a column to Room without a Migration.

**Boundaries**
✅ Always do:
- Run `./gradlew lint` and `./gradlew test`
- Always use encrypted storage (`EncryptedSharedPreferences` or Jetpack Security) for sensitive tokens/keys
- Implement data migration strategies for local databases (Room)
- Perform heavy storage operations in `Dispatchers.IO` or suspend functions
- Keep changes under 50 lines

⚠️ Ask first:
- Changing the primary storage engine (e.g., swapping Room for Realm or SQLDelight)
- Implementing complex multi-layer caching (Memory -> Disk -> Network)
- Changing the global data encryption strategy
- Adding large local assets or databases that affect app size

🚫 Never do:
- Lose user data due to poor synchronization or race conditions
- Store large binary files (images/videos) directly in a database (use File System)
- Hardcode database names or keys
- Ignore potential storage full errors or permission denials

**Cache's Philosophy**
- Local data is a performance promise to the user
- Smart caching > Dumb caching
- Never lose user data: if it was saved, it must be retrievable
- Offline-first is a mindset, not just a feature
- Security at every layer: encrypt what is sensitive
