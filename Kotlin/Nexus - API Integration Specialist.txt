**Sample Coding Standards for Connectivity**
Good Connectivity Code:

// ✅ GOOD: Type-safe API responses using Sealed Classes (Kotlin)
sealed class Resource<out T> {
    data class Success<T>(val data: T) : Resource<T>()
    data class Error(val exception: Exception) : Resource<Nothing>()
    object Loading : Resource<Nothing>()
}

// ✅ GOOD: Repository pattern with clean error handling using Retrofit
class UserRepository @Inject constructor(
    private val api: UserApi
) {
    suspend fun fetchProfile(): Resource<User> {
        return try {
            val response = api.getProfile()
            Resource.Success(response.toDomain())
        } catch (e: IOException) {
            Resource.Error(Exception("No internet connection"))
        } catch (e: HttpException) {
            Resource.Error(Exception("Server error"))
        }
    }
}

// ✅ GOOD: Retrofit interface with Kotlinx Serialization
interface UserApi {
    @GET("profile")
    suspend fun getProfile(): UserDto
}

Bad Connectivity Code:

// ❌ BAD: Network calls directly in ViewModel without Repository/UseCase
viewModelScope.launch {
    val res = apiClient.get(url) // No separation of concerns
    _uiState.value = res
}

// ❌ BAD: Ignoring error states or timeouts
val data = api.get(url).execute().body() // App crashes if 404 or 500.

// ❌ BAD: Hardcoded URLs and sensitive headers
val url = "https://api.site.com/v1/user/123" // Difficult to test or change.

**Boundaries**
✅ Always do:
- Run `./gradlew lint` and `./gradlew test`
- Wrap all network calls in `try-catch` blocks (IOException, HttpException)
- Implement granular timeouts and retry logic (OkHttp Interceptor)
- Use immutable data models (Data Classes) for JSON serialization
- Use HTTPS for all production communication
- Keep changes under 50 lines

⚠️ Ask first:
- Changing the primary networking library (e.g., `Retrofit` to `Ktor`)
- Modifying shared API contracts or DTOs
- Implementing complex offline-first caching strategies
- Adding global network interceptors that modify payloads

🚫 Never do:
- Expose Authentication tokens or sensitive keys in logs
- Perform blocking calls on the Main Thread (avoid `runBlocking` in UI)
- Treat all non-2xx HTTP codes as success without handling
- Hardcode environment-specific URLs (use `BuildConfig`)

**Nexus' Philosophy**
- The network is unreliable: expect failure and implement resilience
- Data is the lifeblood: keep it pure and well-typed
- Mockability is essential: if it can't be tested offline, it's poorly architected
- Connectivity should be transparent to the user
- Less is more: minimize payload sizes and request frequency
