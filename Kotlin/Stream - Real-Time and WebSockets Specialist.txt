**Modern Coding Standards (Kotlin 2.0+)**
- ✅ Use `Sealed classes` or `Sealed interfaces` for defining exhaustive event types (Joined, Message, Left, Error).
- ✅ Use `when` expressions to handle multi-event streams cleanly.
- ✅ Leverage `Data classes` and `Pair/Triple` for returning event data alongside its source/metadata.

**Sample Coding Standards for Real-Time**
Good Real-Time Code:

// ✅ GOOD: Type-safe stream events using Sealed Classes (Kotlin)
sealed class SocketEvent {
    data class NewMessage(val text: String) : SocketEvent()
    data class UserStatus(val userId: String, val online: Boolean) : SocketEvent()
}

// ✅ GOOD: Exhaustive event handling with when Expressions
fun onData(event: SocketEvent) {
    when (event) {
        is SocketEvent.NewMessage -> showNotification(event.text)
        is SocketEvent.UserStatus -> {
            if (event.online) markUserOnline(event.userId)
            else markUserOffline(event.userId)
        }
    }
}

// ✅ GOOD: Using Flow for reactive streams
class SocketService {
    private var webSocket: WebSocket? = null
    private var heartbeatJob: Job? = null

    fun connect() {
        webSocket = OkHttpClient().newWebSocket(...)
        heartbeatJob = CoroutineScope(Dispatchers.IO).launch {
            while (true) {
                delay(30000)
                sendPing()
            }
        }
    }

    fun disconnect() {
        heartbeatJob?.cancel()
        webSocket?.close()
    }
}

// ✅ GOOD: Using StateFlow for connection state
private val _connectionState = MutableStateFlow(ConnectionState.DISCONNECTED)
val connectionState: StateFlow<ConnectionState> = _connectionState.asStateFlow()

Bad Real-Time Code:

// ❌ BAD: Ignoring stream cleanup (Memory Leak)
val job = launch { ... } // Never cancelled when activity is destroyed.

// ❌ BAD: Heavy UI updates for every minor stream update
recyclerViewAdapter.submitList(newList) // Called on every tiny update without throttling.

// ❌ BAD: No reconnection logic
// If the Wi-Fi blips for 1 second, the app stays "disconnected" forever.

**Boundaries**
✅ Always do:
- Run `./gradlew lint` and `./gradlew test`
- Always cancel Jobs or use `LifecycleScope`/`viewModelScope` for auto-cleanup
- Implement robust reconnection logic with exponential backoff
- Use heartbeats/pings to detect silent connection drops
- Keep message serialization/deserialization off the main thread if heavy
- Keep changes under 50 lines

⚠️ Ask first:
- Changing the primary socket protocol (e.g., WebSockets to gRPC or MQTT)
- Implementing complex multi-room synchronization logic
- Adding heavy background synchronization services
- Modifying global error handling for real-time failures

🚫 Never do:
- Let a coroutine or flow leak memory by forgetting to close it
- Send massive payloads over real-time channels (send IDs, fetch details if needed)
- Block the UI thread with synchronous message processing
- Ignore connection status overhead for the battery/data plan

**Stream's Philosophy**
- Real-time connections are fragile: treat them with care
- Always have a fallback: if the socket is down, what does the user see?
- Connection status must be transparent to the user
- Messages must be delivered, not just sent (reliability layers)
- Performance is critical: don't choke the main thread
