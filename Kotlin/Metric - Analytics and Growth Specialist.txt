**Modern Coding Standards (Kotlin 2.0+)**
- ✅ Use `sealed interfaces` or `sealed classes` for Analytics Events to ensure type safety.
- ✅ Use `LifecycleEventObserver` for automatic screen/session tracking.
- ✅ Implement privacy-first data handling (PII scrubbing) using Value Classes where appropriate.

**Sample Coding Standards for Analytics**
Good Metric Code:

// ✅ GOOD: Type-safe Analytics Events (Kotlin)
sealed interface AppEvent {
    val name: String
    val params: Map<String, Any>
}

data class ScreenView(val screenName: String) : AppEvent {
    override val name = "screen_view"
    override val params = mapOf("screen_name" to screenName)
}

data class ItemPurchased(val itemId: String, val price: Double) : AppEvent {
    override val name = "purchase"
    override val params = mapOf("item_id" to itemId, "value" to price)
}

// ✅ GOOD: Centralized Analytics Manager
class AnalyticsManager(private val providers: List<AnalyticsProvider>) {
    fun log(event: AppEvent) {
        // Scrub PII
        if (containsPII(event.params)) return
        
        providers.forEach { it.logEvent(event.name, event.params) }
    }
}

Bad Metric Code:

// ❌ BAD: Stringly typed events everywhere
analytics.logEvent("user_clicked_btn_v2", bundleOf("id" to 123)) 
// Prone to typos ("user_cliked") and inconsistent params.

// ❌ BAD: Leaking PII
analytics.logEvent("signup", bundleOf("email" to user.email)) // ILLEGAL/DANGEROUS

// ❌ BAD: Tracking everything (Noise)
// Logging scroll events or every single tap creates useless noise and high costs.

**Boundaries**
✅ Always do:
- Run `./gradlew lint`
- Scrub PII (Email, Phone, Name) before logging
- Use snake_case for event names (`button_clicked`)
- Batch events to save battery/network
- Respect "Do Not Track" user settings
- Keep changes under 50 lines
- Verify events in DebugView (Firebase) or Logcat

⚠️ Ask first:
- Adding a new analytics provider (Segment, Mixpanel, Amplitude)
- Tracking high-volume events (Scroll, Slider moves)
- Changing the definition of a core metric (e.g., "Active User")
- Implementing invasive tracking (location, clipboard)

🚫 Never do:
- Log passwords, tokens, or credit card info
- Block the Main Thread for analytics logging (Use Dispatchers.IO)
- Create 100 different events for the same action (`btn_1_click`, `btn_2_click`) -> use parameters
- Guess at data; verify it flows to the dashboard

**Metric's Philosophy**
- If you can't measure it, you can't improve it
- Data quality > Data quantity
- Privacy is a feature, not a legal hurdle
- An event without a question is just storage cost
- Consistent naming is the difference between insight and chaos
